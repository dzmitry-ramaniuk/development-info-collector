## 🧭 Рекомендуемые маршруты по разделу

- **Если впервые изучаете очереди сообщений**: начните с Kafka как с самой востребованной платформы, а затем сравните её с ActiveMQ и классическим JMS-подходом
- **Если работаете с интеграцией enterprise-систем**: изучите [ActiveMQ](activemq/README.md), после чего переходите к Kafka для понимания различий между broker-based messaging и event streaming
- **Если интересует production-эксплуатация**: в Kafka уделите особое внимание безопасности, мониторингу, high availability и disaster recovery

## ⚖️ Быстрый выбор: Kafka или ActiveMQ

| Критерий | Kafka | ActiveMQ |
|----------|-------|----------|
| Основной сценарий | Event streaming, аналитические и data-пайплайны | Классический messaging, интеграция приложений, JMS |
| Модель работы | Лог событий с хранением и повторным чтением | Брокер сообщений с доставкой потребителю |
| Когда изучать в первую очередь | Нужны стриминг, replay, высокая пропускная способность | Нужны очереди, topics, JMS и enterprise-интеграция |

Если не уверены, с чего начать:
- **берите Kafka**, если работаете с event-driven архитектурой, аналитикой или микросервисами;
- **берите ActiveMQ**, если в проекте важны JMS, очереди задач и интеграция legacy/enterprise-систем.

## 🧩 Ключевые термины перед стартом

- **Queue** — сообщения читаются потребителями по модели point-to-point.
- **Topic / Pub-Sub** — одно сообщение может быть доставлено нескольким подписчикам.
- **Consumer group** — группа потребителей, совместно читающих поток или набор партиций.
- **Delivery semantics** — гарантии доставки: at-most-once, at-least-once, exactly-once.
