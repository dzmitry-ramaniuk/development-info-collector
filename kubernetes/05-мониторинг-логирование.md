## Мониторинг кластера

### Metrics Server

**Metrics Server** — базовый компонент для сбора метрик ресурсов.

#### Установка:

```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```

#### Использование:

```bash
# Метрики узлов
kubectl top nodes

# Метрики подов
kubectl top pods

# Метрики в конкретном namespace
kubectl top pods -n kube-system

# Сортировка по CPU
kubectl top pods --sort-by=cpu

# Сортировка по памяти
kubectl top pods --sort-by=memory
```

### Kubernetes Dashboard

Веб-интерфейс для управления кластером.

```bash
# Установка
kubectl apply -f https://raw.githubusercontent.com/kubernetes/dashboard/v2.7.0/aio/deploy/recommended.yaml

# Создание Service Account
kubectl create serviceaccount dashboard-admin -n kubernetes-dashboard

# Привязка прав
kubectl create clusterrolebinding dashboard-admin \
  --clusterrole=cluster-admin \
  --serviceaccount=kubernetes-dashboard:dashboard-admin

# Получить токен
kubectl -n kubernetes-dashboard create token dashboard-admin

# Proxy для доступа
kubectl proxy

# URL: http://localhost:8001/api/v1/namespaces/kubernetes-dashboard/services/https:kubernetes-dashboard:/proxy/
```

## Prometheus и Grafana

### Prometheus

**Prometheus** — система мониторинга и alerting.

#### Установка через Helm:

```bash
# Добавить репозиторий
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

# Установить Prometheus
helm install prometheus prometheus-community/prometheus \
  --namespace monitoring \
  --create-namespace
```

#### Prometheus Operator:

```bash
# Установить kube-prometheus-stack (Prometheus + Grafana + Alertmanager)
helm install kube-prometheus prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --create-namespace
```

#### ServiceMonitor

**ServiceMonitor** — декларативная конфигурация для scraping метрик.

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: app-metrics
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: myapp
  endpoints:
  - port: metrics
    interval: 30s
    path: /metrics
```

#### PodMonitor

Для мониторинга подов напрямую.

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PodMonitor
metadata:
  name: app-pods
spec:
  selector:
    matchLabels:
      app: myapp
  podMetricsEndpoints:
  - port: metrics
    interval: 30s
```

#### Prometheus Queries (PromQL)

```promql
# CPU usage
rate(container_cpu_usage_seconds_total[5m])

# Memory usage
container_memory_usage_bytes

# Request rate
rate(http_requests_total[5m])

# Error rate
rate(http_requests_total{status=~"5.."}[5m])

# P95 latency
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))

# Pod restarts
kube_pod_container_status_restarts_total

# Available replicas
kube_deployment_status_replicas_available
```

#### Application Metrics

**Пример с Go:**
```go
import (
    "github.com/prometheus/client_golang/prometheus"
    "github.com/prometheus/client_golang/prometheus/promhttp"
)

var (
    httpRequestsTotal = prometheus.NewCounterVec(
        prometheus.CounterOpts{
            Name: "http_requests_total",
            Help: "Total HTTP requests",
        },
        []string{"method", "endpoint", "status"},
    )
)

func init() {
    prometheus.MustRegister(httpRequestsTotal)
}

func handler(w http.ResponseWriter, r *http.Request) {
    httpRequestsTotal.WithLabelValues(r.Method, r.URL.Path, "200").Inc()
}

func main() {
    http.Handle("/metrics", promhttp.Handler())
    http.ListenAndServe(":8080", nil)
}
```

## Логирование приложений

### Стандартный подход

**Логи в stdout/stderr** — стандарт для контейнеров.

```bash
# Просмотр логов
kubectl logs pod-name

# Follow logs
kubectl logs -f pod-name

# Логи конкретного контейнера
kubectl logs pod-name -c container-name

# Последние N строк
kubectl logs --tail=100 pod-name

# Логи за период
kubectl logs --since=1h pod-name

# Логи предыдущего контейнера (после рестарта)
kubectl logs -p pod-name
```

### Структурированное логирование

**JSON логи** — лучше для парсинга и анализа.

```json
{
  "timestamp": "2024-01-15T10:30:00Z",
  "level": "INFO",
  "message": "Request processed",
  "method": "GET",
  "path": "/api/users",
  "duration_ms": 45,
  "status": 200
}
```

### Centralized Logging

**Архитектура:**
```
Pods (stdout) → Node Agent (Fluentd/Filebeat) → Storage (Elasticsearch) → Visualization (Kibana)
```

### EFK Stack (Elasticsearch, Fluentd, Kibana)

#### 1. Elasticsearch

```bash

helm repo add elastic https://helm.elastic.co
helm install elasticsearch elastic/elasticsearch \
  --namespace logging \
  --create-namespace
```

#### 2. Fluentd

**DaemonSet** для сбора логов с каждого узла.

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
  namespace: logging
spec:
  selector:
    matchLabels:
      name: fluentd
  template:
    metadata:
      labels:
        name: fluentd
    spec:
      serviceAccountName: fluentd
      containers:
      - name: fluentd
        image: fluent/fluentd-kubernetes-daemonset:v1-debian-elasticsearch
        env:
        - name: FLUENT_ELASTICSEARCH_HOST
          value: "elasticsearch.logging.svc.cluster.local"
        - name: FLUENT_ELASTICSEARCH_PORT
          value: "9200"
        volumeMounts:
        - name: varlog
          mountPath: /var/log
        - name: varlibdockercontainers
          mountPath: /var/lib/docker/containers
          readOnly: true
        resources:
          limits:
            memory: 512Mi
          requests:
            cpu: 100m
            memory: 200Mi
      volumes:
      - name: varlog
        hostPath:
          path: /var/log
      - name: varlibdockercontainers
        hostPath:
          path: /var/lib/docker/containers
```

### ELK Stack (Filebeat вместо Fluentd)

**Filebeat** — легковесный log shipper.

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: filebeat
  namespace: logging
spec:
  selector:
    matchLabels:
      name: filebeat
  template:
    metadata:
      labels:
        name: filebeat
    spec:
      containers:
      - name: filebeat
        image: docker.elastic.co/beats/filebeat:8.10.0
        args: [
          "-c", "/etc/filebeat.yml",
          "-e",
        ]
        env:
        - name: ELASTICSEARCH_HOST
          value: elasticsearch.logging.svc.cluster.local
        - name: ELASTICSEARCH_PORT
          value: "9200"
        volumeMounts:
        - name: config
          mountPath: /etc/filebeat.yml
          subPath: filebeat.yml
        - name: varlog
          mountPath: /var/log
        - name: varlibdockercontainers
          mountPath: /var/lib/docker/containers
          readOnly: true
      volumes:
      - name: config
        configMap:
          name: filebeat-config
      - name: varlog
        hostPath:
          path: /var/log
      - name: varlibdockercontainers
        hostPath:
          path: /var/lib/docker/containers
```

### Loki Stack (альтернатива Elasticsearch)

**Loki** — логирование от Grafana Labs, легче и дешевле Elasticsearch.

```bash
# Установка Loki + Promtail + Grafana
helm repo add grafana https://grafana.github.io/helm-charts
helm install loki grafana/loki-stack \
  --namespace logging \
  --create-namespace \
  --set grafana.enabled=true
```

**Преимущества Loki:**
- Меньше требований к ресурсам
- Индексирует только метки (labels), не весь текст
- Интеграция с Grafana
- LogQL — язык запросов похожий на PromQL

```logql
# LogQL примеры
{app="myapp"} |= "error"
{namespace="default"} | json | status >= 500
rate({job="app"}[5m])
```

### Distributed Tracing Example

```python
# Python с OpenTelemetry
from opentelemetry import trace
from opentelemetry.exporter.jaeger.thrift import JaegerExporter
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor

tracer_provider = TracerProvider()
jaeger_exporter = JaegerExporter(
    agent_host_name="jaeger-agent",
    agent_port=6831,
)
tracer_provider.add_span_processor(BatchSpanProcessor(jaeger_exporter))
trace.set_tracer_provider(tracer_provider)

tracer = trace.get_tracer(__name__)

@app.route('/api/user')
def get_user():
    with tracer.start_as_current_span("get_user"):
        # Your code here
        return {"user": "john"}
```

## Alerting

### Prometheus Alertmanager

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: app-alerts
  namespace: monitoring
spec:
  groups:
  - name: app
    interval: 30s
    rules:
    - alert: HighErrorRate
      expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05
      for: 5m
      labels:
        severity: critical
      annotations:
        summary: "High error rate detected"
        description: "Error rate is {{ $value }} for {{ $labels.instance }}"
    
    - alert: PodCrashLooping
      expr: rate(kube_pod_container_status_restarts_total[15m]) > 0
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "Pod is crash looping"
        description: "Pod {{ $labels.pod }} is restarting frequently"
    
    - alert: HighMemoryUsage
      expr: container_memory_usage_bytes / container_spec_memory_limit_bytes > 0.9
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "High memory usage"
        description: "Container {{ $labels.container }} using {{ $value }}% memory"
```

### Alertmanager Configuration

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: alertmanager-config
data:
  config.yml: |
    global:
      resolve_timeout: 5m

      group_wait: 10s
      group_interval: 10s
      repeat_interval: 12h
      receiver: 'slack'
      routes:
      - match:
          severity: critical
        receiver: pagerduty

    - name: 'pagerduty'

      - service_key: 'xxx'
```

## Best Practices

### Логирование

1. **Используйте структурированные логи** (JSON)
2. **Логируйте в stdout/stderr**, не в файлы
3. **Добавляйте контекст**: request ID, user ID, trace ID
4. **Уровни логирования**: DEBUG, INFO, WARN, ERROR
5. **Ротация логов** на уровне node

### Мониторинг

1. **Golden Signals**:
   - **Latency** — время ответа
   - **Traffic** — количество запросов
   - **Errors** — процент ошибок
   - **Saturation** — использование ресурсов

2. **USE Method** (для ресурсов):
   - **Utilization** — процент использования
   - **Saturation** — очередь/перегрузка
   - **Errors** — количество ошибок

3. **RED Method** (для сервисов):
   - **Rate** — запросы в секунду
   - **Errors** — процент ошибок
   - **Duration** — время ответа

### Alerting

1. **Алерты на симптомы**, не на причины
2. **Actionable alerts** — каждый алерт требует действия
3. **Правильная severity**: critical, warning, info
4. **Избегайте alert fatigue** — не слишком много алертов
5. **Документируйте runbooks** для каждого алерта

### Resource Limits

## Вопросы для самопроверки

1. **Что такое Metrics Server и зачем он нужен?**
   - Компонент для сбора метрик ресурсов (CPU, память), используется для HPA и kubectl top

2. **В чём разница между EFK и ELK стеками?**
   - Fluentd vs Filebeat как log shippers; оба используют Elasticsearch + Kibana

3. **Что такое Golden Signals?**
   - Latency, Traffic, Errors, Saturation — ключевые метрики для SRE

4. **Зачем нужен distributed tracing?**
   - Для отслеживания запросов через микросервисы, выявления bottlenecks

5. **Что такое ServiceMonitor в Prometheus?**
   - Декларативная конфигурация для автоматического scraping метрик из сервисов
