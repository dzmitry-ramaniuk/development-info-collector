## Общая архитектура кластера

Kubernetes кластер состоит из двух типов узлов:

### Master Nodes (Control Plane)
Управляющий слой, который контролирует весь кластер:
- Принимает решения о планировании
- Обнаруживает и реагирует на события кластера
- Хранит состояние кластера

### Worker Nodes
Рабочие узлы, которые запускают приложения:
- Запускают поды с контейнерами
- Обеспечивают runtime environment
- Взаимодействуют с Control Plane

```
┌──────────────────────────────────────────────────┐
│           Kubernetes Control Plane               │
│  ┌──────────┐ ┌──────────┐ ┌───────────────┐    │
│  │   API    │ │Controller│ │   Scheduler   │    │
│  │  Server  │ │ Manager  │ │               │    │
│  └─────┬────┘ └────┬─────┘ └───────┬───────┘    │
│        │           │                │            │
│        └───────────┴────────────────┘            │
│                    │                             │
│              ┌─────▼─────┐                       │
│              │   etcd    │                       │
│              └───────────┘                       │
└──────────────────┬───────────────────────────────┘
                   │
         ┌─────────┼─────────┐
         │         │         │
    ┌────▼────┬────▼────┬────▼────┐
    │ Node 1  │ Node 2  │ Node 3  │
    │         │         │         │
    │ Kubelet │ Kubelet │ Kubelet │
    │ Runtime │ Runtime │ Runtime │
    │  Pods   │  Pods   │  Pods   │
    └─────────┴─────────┴─────────┘
```

## Control Plane компоненты

### 1. kube-apiserver

**API Server** — центральный компонент, предоставляющий RESTful API для всех операций.

**Функции:**
- Единая точка входа для всех административных операций
- Валидация и обработка REST запросов
- Обновление объектов в etcd
- Аутентификация и авторизация
- Admission control

**Характеристики:**
- Stateless — может горизонтально масштабироваться
- Общается с etcd для хранения состояния
- Все компоненты взаимодействуют только через API Server

```bash
# Проверка API Server
kubectl cluster-info

# Прямой доступ к API
kubectl proxy
curl http://localhost:8001/api/v1/namespaces/default/pods
```

### 2. etcd

**etcd** — распределённое key-value хранилище для всех данных кластера.

**Что хранит:**
- Конфигурация кластера
- Состояние всех объектов (Pods, Services, Deployments)
- Метаданные
- Секреты и ConfigMaps

**Особенности:**
- Использует Raft consensus алгоритм
- Обеспечивает consistency и high availability
- Критичен для работы кластера
- Требует регулярного backup

**Best Practices:**
- Запускать в количестве 3, 5 или 7 узлов (нечётное число)
- Регулярные бэкапы
- Мониторинг производительности
- SSD диски для production

```bash
# В production etcd обычно работает отдельно
# Backup etcd
etcdctl snapshot save snapshot.db

# Restore
etcdctl snapshot restore snapshot.db
```

### 3. kube-scheduler

**Scheduler** — назначает поды на узлы.

**Процесс планирования:**
1. **Filtering** — отфильтровывает узлы, которые не подходят
   - Недостаточно ресурсов
   - Node selectors не совпадают
   - Taints и tolerations
   - Affinity/anti-affinity правила

2. **Scoring** — ранжирует подходящие узлы
   - Распределение ресурсов
   - Locality preferences
   - Custom scoring plugins

3. **Binding** — привязывает под к выбранному узлу

**Факторы при планировании:**
- **Resource requests** — CPU, память, GPU
- **Node selectors** — метки узлов
- **Affinity/Anti-affinity** — предпочтения размещения
- **Taints and tolerations** — ограничения узлов
- **Pod priority** — приоритет подов

```yaml
# Пример с node selector
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  nodeSelector:
    disktype: ssd
  containers:
  - name: nginx
    image: nginx
```

### 4. kube-controller-manager

**Controller Manager** — запускает различные контроллеры для управления состоянием кластера.

**Основные контроллеры:**

#### Node Controller
- Мониторинг состояния узлов
- Реагирует на отказы узлов
- Eviction подов с недоступных узлов

#### Replication Controller
- Поддерживает нужное количество реплик
- Создаёт/удаляет поды для достижения desired state

#### Endpoints Controller
- Заполняет Endpoints объекты (связь Services и Pods)

#### Service Account & Token Controllers
- Создаёт default accounts и API access tokens

#### Deployment Controller
- Управляет Deployments и ReplicaSets
- Rolling updates и rollbacks

**Принцип работы контроллеров:**
```
┌──────────────┐
│   Observe    │ ← Смотрит текущее состояние
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   Analyze    │ ← Сравнивает с желаемым
└──────┬───────┘
       │
       ▼
┌──────────────┐
│     Act      │ ← Выполняет действия
└──────────────┘
```

### 5. cloud-controller-manager

**Cloud Controller Manager** — интегрируется с облачными провайдерами.

**Функции:**
- Управление нодами (создание/удаление в облаке)
- Управление маршрутами (cloud network routes)
- Управление LoadBalancers
- Управление volumes

**Примеры интеграций:**
- AWS Cloud Controller
- GCP Cloud Controller
- Azure Cloud Controller

## Node компоненты

### 1. kubelet

**Kubelet** — агент, который работает на каждом узле.

**Функции:**
- Запуск и остановка подов
- Мониторинг здоровья подов
- Взаимодействие с container runtime
- Монтирование volumes
- Отчёты о состоянии узла в API Server

**Работа с Pod specs:**
- Получает PodSpecs из API Server
- Гарантирует, что контейнеры запущены и здоровы
- Выполняет health checks (liveness, readiness, startup probes)

```yaml
# Пример health checks
apiVersion: v1
kind: Pod
metadata:
  name: health-check-demo
spec:
  containers:
  - name: app
    image: myapp
    livenessProbe:
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
```

### 2. kube-proxy

**kube-proxy** — сетевой прокси на каждом узле.

**Функции:**
- Реализует Services abstraction
- Настраивает iptables/IPVS rules
- Load balancing между подами
- Обеспечивает network connectivity

**Modes работы:**
1. **iptables mode** (по умолчанию)
   - Использует iptables rules
   - Random load balancing

2. **IPVS mode**
   - Использует Linux IPVS
   - Больше алгоритмов балансировки (round-robin, least-connection)
   - Лучше масштабируется

3. **userspace mode** (legacy)
   - Proxy в user space
   - Медленнее современных методов

```bash
# Посмотреть режим kube-proxy
kubectl logs -n kube-system kube-proxy-xxxxx
```

### 3. Container Runtime

**Container Runtime** — программа для запуска контейнеров.

**Поддерживаемые runtimes:**

#### containerd
- Стандартный runtime в Kubernetes
- Легковесный и эффективный
- Промышленный стандарт

#### CRI-O
- Оптимизирован для Kubernetes
- Минималистичный подход
- Совместим с OCI

#### Docker (deprecated)
- Ранее популярный вариант
- Kubernetes 1.24+ не поддерживает dockershim
- Можно использовать через cri-dockerd

**Container Runtime Interface (CRI):**
```
┌──────────┐
│ Kubelet  │
└────┬─────┘
     │ CRI (gRPC)
     │
┌────▼─────────┐
│   Runtime    │
│ (containerd, │
│   CRI-O)     │
└──────────────┘
```

## Дополнительные компоненты

### CoreDNS

**DNS сервер** для service discovery внутри кластера.

**Функции:**
- Разрешение имён сервисов
- Резолвинг внешних DNS
- Custom DNS записи

```yaml
# Service DNS pattern
<service-name>.<namespace>.svc.cluster.local

# Пример
mysql.default.svc.cluster.local
```

### Metrics Server

**Metrics Server** — сбор метрик ресурсов.

**Назначение:**
- CPU и память подов/узлов
- Horizontal Pod Autoscaling
- kubectl top команды

```bash
# Установка
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

# Использование
kubectl top nodes
kubectl top pods
```

### Ingress Controller

**Ingress Controller** — управляет внешним доступом к сервисам.

**Популярные контроллеры:**
- NGINX Ingress Controller
- Traefik
- HAProxy
- Istio Gateway

```yaml
# Пример Ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
spec:
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app-service
            port:
              number: 80
```

## Сетевая модель Kubernetes

### Требования сетевой модели

**Kubernetes требует:**
1. Все поды могут общаться друг с другом без NAT
2. Все узлы могут общаться со всеми подами без NAT
3. IP адрес пода одинаков изнутри и снаружи

### CNI (Container Network Interface)

**Популярные CNI плагины:**

#### Calico
- Layer 3 networking
- Network policies
- BGP routing

#### Flannel
- Простой overlay network
- Легко настраивается
- VXLAN backend

#### Weave Net
- Mesh network
- Автоматическое обнаружение
- Encryption support

#### Cilium
- eBPF-based networking
- Advanced network policies
- Service mesh features

```bash
# Установка Calico
kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml

# Проверка CNI
kubectl get pods -n kube-system | grep calico
```

### Network Policies

**Network Policies** — firewall правила для подов.

```yaml
# Пример Network Policy
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

## Хранение данных

### Volume Types

**Временные volumes:**
- **emptyDir** — пустая директория, живёт с подом
- **hostPath** — монтирование директории с узла (не рекомендуется)

**Персистентные volumes:**
- **PersistentVolume (PV)** — физический storage
- **PersistentVolumeClaim (PVC)** — запрос на storage
- **StorageClass** — динамическое создание PV

```yaml
# PersistentVolumeClaim
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
  storageClassName: fast-ssd
```

### StorageClass

**StorageClass** — автоматическое provisioning storage.

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3
  iopsPerGB: "10"
```

### CSI (Container Storage Interface)

**CSI drivers** для различных storage систем:
- AWS EBS CSI
- GCE PD CSI
- Azure Disk CSI
- Ceph RBD
- NFS

## Взаимодействие компонентов

### Жизненный цикл создания Pod

```
1. kubectl apply -f pod.yaml
   │
   ▼
2. API Server валидирует и сохраняет в etcd
   │
   ▼
3. Scheduler выбирает узел
   │
   ▼
4. API Server обновляет pod.spec.nodeName
   │
   ▼
5. Kubelet на узле получает PodSpec
   │
   ▼
6. Kubelet инструктирует runtime запустить контейнеры
   │
   ▼
7. Pod запущен и работает
```

### Коммуникация компонентов

**Все компоненты общаются через API Server:**
- Никто не пишет напрямую в etcd
- Все изменения идут через валидацию API
- API Server — единственная точка доступа к состоянию

**Watch механизм:**
- Компоненты подписываются на изменения
- API Server отправляет события
- Контроллеры реагируют на изменения

## Вопросы для самопроверки

1. **Какова роль etcd в Kubernetes?**
   - Распределённое хранилище всего состояния кластера
   - Критичный компонент, требует регулярных backup

2. **Что делает kube-scheduler?**
   - Назначает поды на узлы на основе ресурсов, constraints и policies

3. **Какие режимы работы есть у kube-proxy?**
   - iptables (по умолчанию), IPVS, userspace (legacy)

4. **Зачем нужен kubelet?**
   - Агент на каждом узле, запускает поды и мониторит их здоровье

5. **Что такое CNI?**
   - Container Network Interface — плагины для сетевого взаимодействия подов
