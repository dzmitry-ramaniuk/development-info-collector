## ConfigMaps

**ConfigMap** — хранение конфигурационных данных в виде key-value пар.

### Создание ConfigMap

#### Из literal значений:

```bash
kubectl create configmap app-config \
  --from-literal=database_url=postgres://db:5432 \
  --from-literal=cache_enabled=true
```

#### Из файла:

```bash
# app.properties
database_url=postgres://db:5432
cache_enabled=true
log_level=info

kubectl create configmap app-config --from-file=app.properties
```

#### Из директории:

```bash
kubectl create configmap app-config --from-file=./config/
```

#### Из YAML манифеста:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database_url: "postgres://db:5432"
  cache_enabled: "true"
  log_level: "info"
  app.properties: |
    database_url=postgres://db:5432
    cache_enabled=true
    log_level=info
```

### Использование ConfigMap

#### Как environment variables:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: myapp:1.0
    env:
    # Отдельные ключи
    - name: DATABASE_URL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: database_url
    # Все ключи из ConfigMap
    envFrom:
    - configMapRef:
        name: app-config
```

#### Как volume:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: myapp:1.0
    volumeMounts:
    - name: config
      mountPath: /etc/config
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
```

#### Конкретные файлы из ConfigMap:

```yaml
volumes:
- name: config
  configMap:
    name: app-config
    items:
    - key: app.properties
      path: application.properties
```

### Обновление ConfigMap

```bash
# Редактировать
kubectl edit configmap app-config

# Или применить новый манифест
kubectl apply -f configmap.yaml
```

> **Важно**: Поды не перезапускаются автоматически при изменении ConfigMap. Нужно перезапустить поды вручную или использовать Rolling Update deployment.

## Secrets

**Secret** — безопасное хранение чувствительных данных (пароли, токены, ключи).

### Типы Secrets

1. **Opaque** (generic) — произвольные key-value данные
2. **kubernetes.io/service-account-token** — Service Account token
3. **kubernetes.io/dockerconfigjson** — Docker registry credentials
4. **kubernetes.io/tls** — TLS сертификаты
5. **kubernetes.io/ssh-auth** — SSH ключи
6. **kubernetes.io/basic-auth** — Basic authentication

### Создание Secrets

#### Generic Secret:

```bash
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password=secretpass123
```

#### Из файлов:

```bash
kubectl create secret generic tls-secret \
  --from-file=tls.crt=server.crt \
  --from-file=tls.key=server.key
```

#### Docker Registry Secret:

```bash
kubectl create secret docker-registry regcred \
  --docker-server=docker.io \
  --docker-username=myuser \
  --docker-password=mypassword \
  --docker-email=myemail@example.com
```

#### TLS Secret:

```bash
kubectl create secret tls tls-secret \
  --cert=path/to/tls.cert \
  --key=path/to/tls.key
```

#### YAML манифест:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  username: YWRtaW4=      # base64 encoded "admin"
  password: c2VjcmV0cGFzcw==  # base64 encoded "secretpass"
```

```bash
# Base64 encoding
echo -n 'admin' | base64
# YWRtaW4=

# Base64 decoding
echo 'YWRtaW4=' | base64 --decode
# admin
```

#### StringData (без base64):

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
stringData:
  username: admin
  password: secretpass
```

### Использование Secrets

#### Как environment variables:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: myapp:1.0
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password
```

#### Как volume (более безопасно):

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: myapp:1.0
    volumeMounts:
    - name: secret-volume
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secret-volume
    secret:
      secretName: db-secret
```

Файлы в контейнере:
```
/etc/secrets/username
/etc/secrets/password
```

#### ImagePullSecrets:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: private-app
spec:
  imagePullSecrets:
  - name: regcred
  containers:
  - name: app
    image: private-registry.com/myapp:1.0
```

### Безопасность Secrets

**Best Practices:**

1. **Используйте RBAC** для контроля доступа
2. **Encrypt at rest** — включите шифрование etcd
3. **Избегайте env variables** — предпочитайте volumes
4. **Используйте внешние Secret managers** (HashiCorp Vault, AWS Secrets Manager)
5. **Не коммитьте в Git** — secrets должны быть в .gitignore

```yaml
# Включение шифрования etcd
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
    - aescbc:
        keys:
        - name: key1
          secret: <base64 encoded secret>
    - identity: {}
```

### External Secrets Operator

Интеграция с внешними Secret managers:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-secret
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: db-secret
  data:
  - secretKey: username
    remoteRef:
      key: database/credentials
      property: username
```

## Environment Variables

### Типы переменных окружения

#### 1. Статические значения:

```yaml
env:
- name: ENV_TYPE
  value: "production"
- name: PORT
  value: "8080"
```

#### 2. Из ConfigMap:

```yaml
env:
- name: CONFIG_VALUE
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: some-key
```

#### 3. Из Secret:

```yaml
env:
- name: API_KEY
  valueFrom:
    secretKeyRef:
      name: api-secret
      key: api-key
```

#### 4. Field references (Pod metadata):

```yaml
env:
- name: POD_NAME
  valueFrom:
    fieldRef:
      fieldPath: metadata.name
- name: POD_NAMESPACE
  valueFrom:
    fieldRef:
      fieldPath: metadata.namespace
- name: POD_IP
  valueFrom:
    fieldRef:
      fieldPath: status.podIP
- name: NODE_NAME
  valueFrom:
    fieldRef:
      fieldPath: spec.nodeName
```

#### 5. Resource limits:

```yaml
env:
- name: MEMORY_LIMIT
  valueFrom:
    resourceFieldRef:
      containerName: app
      resource: limits.memory
      divisor: 1Mi
```

### EnvFrom — массовый импорт

```yaml
envFrom:
- configMapRef:
    name: app-config
- secretRef:
    name: app-secrets
- prefix: DB_
  configMapRef:
    name: database-config
```

## Volumes

### Типы Volumes

#### 1. emptyDir
Временное хранилище, живёт с подом.

```yaml
volumes:
- name: cache
  emptyDir: {}

# In-memory emptyDir
volumes:
- name: cache
  emptyDir:
    medium: Memory
    sizeLimit: 1Gi
```

#### 2. hostPath
Монтирует директорию с узла (не рекомендуется для production).

```yaml
volumes:
- name: logs
  hostPath:
    path: /var/log/myapp
    type: DirectoryOrCreate
```

#### 3. configMap

```yaml
volumes:
- name: config
  configMap:
    name: app-config
```

#### 4. secret

```yaml
volumes:
- name: secret
  secret:
    secretName: app-secret
```

#### 5. persistentVolumeClaim

```yaml
volumes:
- name: data
  persistentVolumeClaim:
    claimName: mysql-pvc
```

#### 6. projected
Комбинирует несколько источников в один volume.

```yaml
volumes:
- name: all-in-one
  projected:
    sources:
    - secret:
        name: db-secret
    - configMap:
        name: app-config
    - downwardAPI:
        items:
        - path: "labels"
          fieldRef:
            fieldPath: metadata.labels
```

## Persistent Volumes

### PersistentVolume (PV)

**PV** — физическое хранилище в кластере.

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-example
spec:
  capacity:
    storage: 10Gi
  volumeMode: Filesystem
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: fast
  hostPath:
    path: /mnt/data
```

**Access Modes:**
- **ReadWriteOnce (RWO)** — один узел, read-write
- **ReadOnlyMany (ROX)** — множество узлов, read-only
- **ReadWriteMany (RWX)** — множество узлов, read-write
- **ReadWriteOncePod (RWOP)** — один под, read-write (K8s 1.22+)

**Reclaim Policy:**
- **Retain** — данные сохраняются после удаления PVC
- **Delete** — удаляет PV и данные
- **Recycle** (deprecated) — очистка данных для повторного использования

### PersistentVolumeClaim (PVC)

**PVC** — запрос на хранилище.

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
  storageClassName: fast
  selector:
    matchLabels:
      type: database
```

### Использование PVC в Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: mysql
spec:
  containers:
  - name: mysql
    image: mysql:8.0
    volumeMounts:
    - name: data
      mountPath: /var/lib/mysql
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: mysql-pvc
```

### Volume Expansion

```yaml
# Изменить размер PVC
spec:
  resources:
    requests:
      storage: 20Gi  # Было 10Gi
```

```bash
# Проверить поддержку expansion
kubectl get storageclass fast -o yaml | grep allowVolumeExpansion
```

## Storage Classes

**StorageClass** — автоматическое создание PV (Dynamic Provisioning).

### Примеры StorageClass

#### AWS EBS:

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  iopsPerGB: "10"
  encrypted: "true"
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
reclaimPolicy: Delete
```

#### GCE PD:

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: pd.csi.storage.gke.io
parameters:
  type: pd-ssd
  replication-type: regional-pd
```

#### Local Storage:

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-storage
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
```

### Default StorageClass

```yaml
metadata:
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
```

```bash
# Установить default StorageClass
kubectl patch storageclass fast-ssd -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
```

### Volume Binding Modes

**Immediate** — PV создаётся сразу при создании PVC

**WaitForFirstConsumer** — PV создаётся при создании первого пода (лучше для topology constraints)

## Best Practices

### ConfigMaps и Secrets

1. **Разделение по окружениям**
```bash
config-dev
config-staging
config-production
```

2. **Версионирование**
```yaml
metadata:
  name: app-config-v2
```

3. **Используйте immutable ConfigMaps/Secrets** (K8s 1.21+)
```yaml
immutable: true
```

4. **Избегайте больших ConfigMaps**
   - Лимит: 1 MB
   - Для больших файлов используйте volumes

### Secrets Security

1. **RBAC ограничения**
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list"]
  resourceNames: ["specific-secret"]
```

2. **Audit logging**
```yaml
# Аудит доступа к secrets
audit.k8s.io/v1
```

3. **Используйте Sealed Secrets**
```bash
# Шифрование секретов для Git
kubeseal -o yaml < secret.yaml > sealed-secret.yaml
```

### Storage

1. **Используйте PVC вместо PV напрямую**

2. **Backup Persistent Volumes**
```bash
# Velero для backup
velero backup create my-backup --include-resources pvc,pv
```

3. **Мониторинг использования**
```bash
kubectl top pods
kubectl describe pvc
```

4. **Используйте volumeMode правильно**
   - **Filesystem** — для обычных приложений
   - **Block** — для БД с прямым доступом к устройству

## Вопросы для самопроверки

1. **В чём разница между ConfigMap и Secret?**
   - ConfigMap для обычной конфигурации, Secret для чувствительных данных (base64, encryption at rest)

2. **Почему volumes предпочтительнее env для Secrets?**
   - Env variables могут попасть в логи, volumes более безопасны

3. **Что такое Dynamic Provisioning?**
   - Автоматическое создание PV через StorageClass при создании PVC

4. **Какие Access Modes существуют?**
   - RWO (ReadWriteOnce), ROX (ReadOnlyMany), RWX (ReadWriteMany), RWOP (ReadWriteOncePod)

5. **Зачем нужен WaitForFirstConsumer binding mode?**
   - Для учёта topology constraints при создании volume
