## Зачем отдельно проектировать метрики

Метрики — это не «добавим потом», а часть архитектуры production-системы. Если заранее не определить, что измерять и как интерпретировать сигнал, то при инциденте команда видит шум вместо причины.

> Практика: сначала фиксируем пользовательские и бизнес-критичные сценарии, затем для каждого сценария определяем SLI и алерты.

## Подходы к метрикам: от ad-hoc к SLI/SLO

Эволюция зрелости обычно выглядит так:

1. **Ad-hoc monitoring** — смотрим CPU/RAM и отдельные графики без явной связи с качеством сервиса.
2. **Service dashboards** — добавляем latency, error rate, traffic по каждому сервису.
3. **SLI/SLO-driven** — определяем измеримые цели качества и error budget.
4. **Product-level observability** — связываем технические метрики с бизнес-воронкой (регистрация, оплата, доставка).

Для backend-систем удобно использовать:
- **RED** (Rate, Errors, Duration) для HTTP/gRPC/API;
- **USE** (Utilization, Saturation, Errors) для инфраструктуры и ресурсов;
- **Golden Signals** как универсальный каркас для дашбордов и on-call.

## Какие метрики собирать в первую очередь

Ниже — минимальный базовый набор для Java-сервиса:

- **Трафик**: RPS, throughput, размер очереди запросов.
- **Ошибки**: 4xx/5xx, исключения по типам, доля timeout.
- **Задержка**: p50/p95/p99 latency по endpoint/операции.
- **Ресурсы JVM**: heap/non-heap, GC pause time, thread states.
- **Интеграции**: latency и error rate внешних API, БД, брокеров.
- **Бизнес-метрики**: успешные заказы, конверсия, время завершения ключевого сценария.

## Elastic Stack для метрик и логов

**Elastic Stack** чаще всего используют, когда нужен единый контур для логов, метрик и поиска по инцидентам.

Базовые компоненты:

- **Elasticsearch** — хранение и поиск временных рядов/документов.
- **Kibana** — дашборды, визуализация и ad-hoc анализ.
- **Beats / Elastic Agent** — сбор телеметрии с хостов/контейнеров.
- **Logstash** (опционально) — обогащение и маршрутизация потока данных.
- **APM Server** — сбор APM-сигналов (трейсы, метрики приложений).

Когда Elastic Stack особенно уместен:

- нужна мощная полнотекстовая аналитика и корреляция «логи + метрики + trace-контекст»;
- уже есть экспертиза команды в Elastic ecosystem;
- важна единая платформа с гибкими запросами и визуализацией.

Ограничения и компромиссы:

- стоимость хранения и масштабирования может быстро расти;
- требуется дисциплина по schema/data lifecycle (индексы, retention, rollover);
- нужна регулярная оптимизация запросов и шардирования.

## Open-source стек наблюдаемости

Если нужен максимально открытый и модульный подход, часто выбирают связку:

- **Prometheus** — сбор и хранение метрик;
- **Alertmanager** — маршрутизация и дедупликация алертов;
- **Grafana** — дашборды и алертинг поверх разных источников;
- **Loki** — централизованные логи (дешевле full-text сценариев);
- **Tempo** или **Jaeger** — распределённый tracing;
- **OpenTelemetry Collector** — единая точка приёма/преобразования телеметрии.

Плюсы open-source стека:

- высокая переносимость и отсутствие vendor lock-in;
- гибкая компоновка под конкретную нагрузку;
- большой выбор интеграций для Kubernetes и облаков.

Минусы:

- больше операционной ответственности на команду;
- нужно проектировать federation/long-term storage/high availability заранее;
- сложнее обеспечить «из коробки» единый пользовательский опыт, чем в монолитных платформах.

## Java: фреймворки и библиотеки для метрик

Практически полезные инструменты для Java-приложений:

- **Micrometer** — стандарт де-факто для метрик в Spring Boot, единый facade над registry.
- **Spring Boot Actuator** — готовые endpoint-ы health/info/metrics и интеграция с Micrometer.
- **OpenTelemetry Java SDK + javaagent** — трейсинг и метрики с автоинструментированием.
- **Prometheus Java client** — нативная экспозиция метрик в формате Prometheus.
- **Dropwizard Metrics** — проверенная библиотека, часто встречается в legacy/infra-проектах.
- **JMX Exporter** — экспорт JMX/JVM метрик в Prometheus-совместимый формат.

Рекомендации по выбору:

1. Для **Spring Boot** чаще всего: Actuator + Micrometer + OpenTelemetry.
2. Для «чистой» Java без Spring: OpenTelemetry SDK/agent + Prometheus client.
3. Для миграции legacy: постепенно унифицировать метрики через Micrometer bridge/registry и убрать дублирующие счётчики.

## Рекомендуемые архитектурные подходы

- **Единая схема именования метрик**: сервис, операция, результат, окружение.
- **Кардинальность под контролем**: избегайте tag-ов с userId/orderId/requestId.
- **SLO-ориентированные алерты**: меньше alert fatigue, больше сигнала.
- **Runbook рядом с алертом**: у каждого критичного алерта должна быть инструкция действий.
- **Инструментирование по слоям**: edge/API, бизнес-операции, интеграции, инфраструктура.
- **Retention по ценности**: краткосрочные high-resolution данные + долгосрочные агрегаты.

## Практический минимальный план внедрения

1. Выберите 2–3 критичных user journey и зафиксируйте их SLI/SLO.
2. Подключите JVM + HTTP метрики и базовый dashboard RED/USE.
3. Настройте 5–7 ключевых алертов (ошибки, latency, saturation).
4. Добавьте трассировку для внешних интеграций (БД, брокер, внешние API).
5. Введите review метрик в Definition of Done для новых фич.
6. Раз в спринт удаляйте шумные/неполезные алерты и дашборды.

## Типичные ошибки

- чрезмерная детализация метрик на старте без понятной цели;
- «метрики ради метрик» без связи с SLO и бизнес-риском;
- высокая кардинальность лейблов, которая ломает storage/query;
- отсутствие владельца дашборда и алертов;
- отсутствие тестов наблюдаемости (проверок, что метрики действительно публикуются).
