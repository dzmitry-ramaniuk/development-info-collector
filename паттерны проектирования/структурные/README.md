# Структурные паттерны

Структурные паттерны (Structural Patterns) — это паттерны проектирования, которые определяют способы компоновки объектов и классов в более крупные структуры, сохраняя при этом гибкость и эффективность этих структур.

   - Примеры из Java Collections

2. [Bridge (Мост)](02-bridge.md)

   - Примеры в GUI-фреймворках

## 🎯 Как использовать

### Для начинающих
Начните с паттернов Adapter и Decorator — они наиболее интуитивны и часто встречаются в стандартных библиотеках Java. Затем изучите Facade, который помогает проектировать понятные API.

### Для подготовки к собеседованиям
Обязательно знайте:
- **Decorator vs Inheritance**: Когда использовать композицию вместо наследования
- **Proxy**: Различные типы прокси и их применение (особенно в Spring)
- **Adapter vs Bridge**: В чем разница между этими похожими паттернами
- Примеры из JDK: `InputStreamReader` (Adapter), `BufferedReader` (Decorator), `String.valueOf()` (Flyweight)

### Для опытных разработчиков
Изучите нюансы:
- Использование динамических прокси в Java и их ограничения
- Применение Decorator в архитектуре middleware и filters
- Flyweight в контексте производительности и управления памятью
- Комбинирование структурных паттернов (например, Composite + Decorator)

## 💡 Рекомендации

- **Adapter**: Предпочитайте композицию (адаптер объектов) наследованию (адаптер классов) для большей гибкости
- **Decorator**: Используйте для добавления функциональности без изменения исходного класса
- **Facade**: Создавайте фасады для упрощения работы со сложными библиотеками и подсистемами
- **Proxy**: В Spring большинство AOP-функциональности реализовано через динамические прокси

## ⚠️ Важные замечания

> **Decorator и производительность**: Множественные уровни декораторов могут создавать длинные цепочки вызовов, что может повлиять на производительность.

> **Proxy в Spring**: Spring использует JDK Dynamic Proxy для интерфейсов и CGLIB для классов. Это важно понимать при работе с AOP и транзакциями.

> **Flyweight и неизменяемость**: Flyweight лучше всего работает с неизменяемыми объектами, чтобы избежать проблем с разделяемым состоянием.

> **Composite и рекурсия**: При работе с древовидными структурами будьте внимательны к глубине рекурсии и возможному переполнению стека.
