## Зачем тюнить JVM и когда это действительно нужно

Тюнинг JVM нужен не для «максимальных бенчмарков», а для соблюдения SLA по задержкам, стабильности и стоимости инфраструктуры.

> Хорошая стратегия: сначала измеряйте, потом меняйте флаги, и только после этого сравнивайте результат на одинаковой нагрузке.

Когда тюнинг точно нужен:
- есть регулярные всплески latency (p95/p99);
- контейнер упирается в лимиты памяти и перезапускается;
- GC-паузы влияют на бизнес-операции;
- throughput сервиса «плавает» при росте трафика.

## Метрики, с которых начинается диагностика

Минимальный набор метрик для JVM-сервиса:

| Группа | Что смотреть | Почему это важно |
|---|---|---|
| Память | `heap used/max`, old gen occupancy, metaspace, direct memory | Позволяет увидеть утечки, неверный размер heap и давление аллокатора |
| GC | частота коллекций, длительность пауз, allocation rate, promotion rate | Показывает, где теряется latency и процессор |
| CPU | загрузка процесса JVM, system load, steal time | Отделяет проблему GC от нехватки CPU |
| Потоки | количество runnable/blocked потоков, очереди пулов | Помогает найти contention и saturation |
| Ошибки | OOM, timeout, рост 5xx | Связывает инфраструктурные симптомы с пользовательским эффектом |

## Базовый production-набор JVM-флагов

Ниже стартовый шаблон (чаще всего для Java 17+ и серверных приложений):

```bash
java \
  -Xms2g -Xmx2g \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/log/app/heapdump.hprof \
  -Xlog:gc*:file=/var/log/app/gc.log:time,uptime,level,tags \
  -XX:+ExitOnOutOfMemoryError \
  -jar app.jar
```

Практические замечания:
1. Для предсказуемости чаще задают `-Xms` равным `-Xmx`.
2. G1 — разумный default для большинства backend-нагрузок; ZGC/Shenandoah рассматривайте при сверхжёстких требованиях к паузам.
3. Логи GC должны быть включены всегда: без них тюнинг превращается в догадки.

## Подход к тюнингу GC без "магии"

Пошаговый подход:
1. Зафиксируйте baseline (нагрузка, p95/p99, CPU, память, текущие флаги).
2. Снимите GC-логи на типичной и пиковой нагрузке.
3. Меняйте не более 1–2 параметров за итерацию.
4. Сравнивайте результаты на одинаковом профиле трафика.
5. Оставляйте только изменения, которые улучшили SLA или стоимость.

Частая ошибка — сразу менять десятки флагов из случайных статей. Это ухудшает воспроизводимость и усложняет откат.

## Инструменты мониторинга и профилирования

Типичный набор инструментов:
- **Micrometer + Prometheus + Grafana** — метрики приложения и JVM-дашборды;
- **jcmd / jstat / jmap / jstack** — базовая диагностика на узле;
- **Java Flight Recorder (JFR)** — low-overhead профилирование в production;
- **async-profiler** — CPU/alloc профили и flamegraph;
- **APM (New Relic, Datadog, Elastic APM и др.)** — сквозная корреляция latency, трейсов и инфраструктуры.

Пример полезных команд:

```bash
# Список JVM-процессов
jcmd -l

# Краткая сводка по GC
jstat -gc <pid> 1s 10

# Thread dump для анализа блокировок/зависаний
jcmd <pid> Thread.print > thread-dump.txt

# Запуск JFR на 5 минут
jcmd <pid> JFR.start name=on_demand duration=5m filename=recording.jfr
```

## Типовые проблемы и быстрые действия

1. **Частые stop-the-world паузы**
   - Проверьте allocation rate и размер young generation.
   - Снизьте churn объектов (кэшируйте переиспользуемые структуры, уменьшайте лишние аллокации).

2. **OutOfMemoryError: Java heap space**
   - Снимите heap dump и проверьте доминирующие retainers.
   - Убедитесь, что рост памяти не вызван неограниченными кэшами/коллекциями.

3. **Высокий CPU, но GC не выглядит критичным**
   - Снимите CPU profile (JFR или async-profiler).
   - Проверьте горячие методы, сериализацию, regex, блокировки.

4. **Проблемы только в контейнерах**
   - Проверьте лимиты памяти/CPU и соответствие `-Xmx` этим лимитам.
   - Сверьте requests/limits, чтобы избежать агрессивного throttling.

## Практический чек-лист для production

- Включены GC-логи и ротация файлов логов.
- Настроены heap dump при OOM и доступ к артефактам расследования.
- На дашбордах есть p50/p95/p99, GC pause, allocation rate, CPU, потоки.
- Есть runbook: что делать при OOM, росте latency и saturation пулов.
- Любой тюнинг подтверждается нагрузочным сравнением «до/после».
