## Введение

**Amazon Web Services (AWS)** — крупнейшая облачная платформа с десятками инфраструктурных и прикладных сервисов. Для backend-разработчика AWS обычно становится средой, где живут API, базы данных, очереди, мониторинг и CI/CD-пайплайны.

Цель этого материала — сформировать фундамент: как устроена инфраструктура AWS, как безопасно выдавать доступы, какие сервисы чаще всего используются и как не совершать типичные ошибки в production.

## Глобальная инфраструктура AWS

AWS строится вокруг иерархии географических объектов:

- **Region (регион)** — отдельная географическая область (например, `eu-central-1`).
- **Availability Zone (AZ)** — изолированный дата-центр внутри региона.
- **Edge Locations** — точки присутствия для CDN и ускорения доставки контента.

Практически важно:

- Размещайте критичные компоненты минимум в **2 AZ** для отказоустойчивости.
- Выбирайте регион ближе к пользователям, чтобы снизить latency.
- Учитывайте требования по хранению данных (compliance, residency).

## Shared Responsibility Model

Модель разделяет ответственность между AWS и клиентом:

- **AWS отвечает за “security OF the cloud”**: физическая безопасность дата-центров, гипервизор, базовая инфраструктура.
- **Клиент отвечает за “security IN the cloud”**: настройку IAM, сетевые правила, шифрование данных, патчи ОС (для EC2), конфигурацию приложений.

Вывод для инженера: «облако» не означает «безопасно по умолчанию». Без правильной конфигурации IAM и сетей даже managed-сервисы остаются уязвимыми.

## IAM: управление доступом

**IAM (Identity and Access Management)** — ядро безопасности в AWS.

Базовые сущности:

- **User** — пользователь (обычно для людей и консольного доступа).
- **Group** — группа пользователей с общими правами.
- **Role** — роль, которую принимает сервис/приложение/пользователь временно.
- **Policy** — JSON-документ с разрешениями (`Allow`/`Deny`) на действия и ресурсы.

Ключевые best practices:

1. Принцип **least privilege** — выдавайте минимум необходимых прав.
2. Избегайте долгоживущих access keys для приложений.
3. Для workload’ов используйте **IAM Roles** (например, role для EC2/ECS/Lambda).
4. Включайте MFA для интерактивного доступа пользователей.
5. Разделяйте аккаунты по окружениям: `dev`, `stage`, `prod`.

## Типовые архитектурные сценарии

1. **Классический backend**:
   - ALB → EC2/ECS → RDS
   - S3 для статики и бэкапов
   - CloudWatch + CloudTrail для наблюдаемости и аудита

2. **Serverless API**:
   - API Gateway → Lambda → DynamoDB
   - S3 для файлов
   - EventBridge/SQS для асинхронных задач

3. **Гибридный подход**:
   - Основной API в контейнерах (ECS/EKS)
   - Фоновые задачи через Lambda
   - Кэш через ElastiCache

## Практические советы

- Начните с **Infrastructure as Code** (Terraform, AWS CDK или CloudFormation), а не с ручных кликов в консоли.
- Делайте теги (`Environment`, `Service`, `Owner`, `CostCenter`) обязательными для ресурсов.
- Включайте бюджеты и алерты по расходам в AWS Budgets на раннем этапе.
- Планируйте резервное копирование и регулярное восстановление (backup restore drills).
- Проверяйте права IAM через policy simulation перед выкладкой в production.

## Распространённые ошибки

- Использование одного AWS-аккаунта для всех окружений.
- Широкие права вида `Action: "*", Resource: "*"`.
- Публичные S3-bucket’ы без явной необходимости.
- Отсутствие алертов по CPU/памяти/ошибкам/стоимости.
- Отсутствие multi-AZ для критичных компонентов.
