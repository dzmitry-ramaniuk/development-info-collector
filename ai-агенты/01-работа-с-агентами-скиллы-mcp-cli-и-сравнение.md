## Введение

AI-агент в разработке — это не «магический автокомплит», а исполнитель задач по заданному процессу: он читает контекст, применяет инструкции, использует инструменты (файлы, терминал, внешние источники), затем выдаёт результат.

Ключевая идея: чем лучше вы задали **цель**, **границы** и **критерии приёмки**, тем стабильнее и полезнее результат.

## Как эффективно работать с AI-агентами

### Базовый цикл работы

1. Сформулируйте задачу через результат, а не через абстракцию.
2. Уточните ограничения: стиль кода, формат ответа, какие файлы можно менять.
3. Дайте критерии готовности: тесты, проверки линтером, обновление документации.
4. Запросите короткий план перед реализацией.
5. Проведите ревью изменений и итеративно уточняйте.

### Что давать агенту на вход

- Структуру проекта (где корневой README, где модули, где тесты).
- Локальные правила (AGENTS.md, CONTRIBUTING.md, style guide).
- Явный контекст версии (язык, фреймворк, окружение).
- Примеры «как правильно» из вашего репозитория.

Практический шаблон постановки задачи:

```text
Цель: добавить раздел X в документацию.
Ограничения: не менять существующие разделы Y, использовать русский язык.
Критерии готовности: обновлены README, проверены внутренние ссылки.
Формат отчёта: summary + список проверок командами.
```

## Скиллы: как конфигурировать и поддерживать

**Скилл** — это повторно используемая инструкция/процедура для агента под узкий тип задач (например: «подготовить релизный changelog», «создать PR по шаблону», «проверить структуру раздела в docs»).

### Минимальная структура скилла

```text
my-skill/
  SKILL.md
  templates/
  scripts/
  references/
```

Где:
- `SKILL.md` — когда применять скилл, шаги выполнения, ограничения.
- `templates/` — заготовки для повторяемого результата.
- `scripts/` — автоматизация рутинных шагов.
- `references/` — узкоспециализированные справки.

### Правила качественного скилла

- Один скилл — одна цель (без «универсальных комбайнов»).
- Чёткие триггеры: когда использовать, когда не использовать.
- Проверяемый финал: какие артефакты и проверки должны появиться.
- Минимум неоднозначности в формулировках.
- Версионирование: фиксируйте изменения скилла в истории репозитория.

## MCP: зачем нужен и как подключать

**MCP (Model Context Protocol)** помогает агенту безопасно работать с внешними источниками контекста и инструментами: файлами, базами, API, системами задач.

Типовая ценность MCP:
- меньше копипаста в промпт;
- доступ к «живым» данным вместо устаревших сниппетов;
- контролируемые интеграции через серверы MCP.

Базовый процесс подключения:
1. Поднимите/подключите MCP-сервер для нужной системы (например, Git, issue-tracker, docs).
2. Опишите ресурсы и шаблоны ресурсов, которые агент может читать.
3. Ограничьте область доступа по принципу минимальных привилегий.
4. Протестируйте сценарий чтения контекста и выполнения одной безопасной операции.

## CLI-подход: когда удобнее терминал, а не чат

Работа через CLI особенно эффективна, когда нужно:
- выполнить последовательность команд в репозитории;
- быстро проверить состояние (тесты, линтеры, git diff);
- получить воспроизводимый лог действий.

Примеры полезных команд:

```bash
# Проверить изменённые файлы
rg --files

# Посмотреть краткий статус репозитория
git status --short

# Найти ссылки на конкретный документ
rg "01-работа-с-агентами"
```

## Copilot vs Claude vs Codex: основные различия

Ниже — практичное сравнение (без привязки к маркетингу):

| Инструмент | Сильные стороны | Ограничения/риски | Когда выбирать |
|---|---|---|---|
| **GitHub Copilot** | Быстрое автодополнение и генерация кода прямо в IDE | Может не держать длинный контекст архитектуры | Когда нужно ускорить ежедневный coding flow в редакторе |
| **Claude** | Сильная работа с длинным текстом, анализ требований, аккуратные объяснения | Реализация в коде может требовать более жёстких технических рамок | Когда важны анализ, проектирование, документация, RFC |
| **Codex (agentic CLI/IDE workflow)** | Автономная работа по шагам: чтение файлов, правки, запуск команд и проверок | Требует точных инструкций и критериев приёмки, иначе возможны лишние изменения | Когда нужен полуавтоматический delivery-задач в репозитории |

## Практические сценарии выбора инструмента

1. **Нужно быстро писать много однотипного кода в IDE** → начните с Copilot.
2. **Нужно разобрать объёмные требования и выстроить аргументацию** → Claude.
3. **Нужно внести изменения в проект, прогнать проверки и подготовить PR** → Codex.
4. **Комбинированный workflow**: анализ в Claude, реализация с проверками в Codex, локальная скорость набора через Copilot.

## Типичные ошибки и как их избежать

- Ошибка: «Сделай хорошо» без критериев.
  - Решение: задавайте DoD (Definition of Done) в явном виде.
- Ошибка: отсутствие локальных правил проекта.
  - Решение: перед запуском задачи передайте AGENTS/README/стайлгайды.
- Ошибка: попытка одним промптом решить слишком большую задачу.
  - Решение: дробите на этапы и фиксируйте результат каждого шага.
- Ошибка: слепое принятие результата агента.
  - Решение: обязательный code review и запуск проверок.
